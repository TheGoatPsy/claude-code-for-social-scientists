# Skill Title Here

## When to use

Name the situations that should activate the skill, in one short paragraph. State what the user is holding at that moment and what they want from it. If a neighboring project skill handles an adjacent situation, say which one and when to prefer it.

## Inputs

- The artifact the skill operates on.
- The constraints that change the outcome, such as language scope, venue rules, or deadlines.
- What the user must authorize or provide themselves.

## Workflow

1. Start from a concrete first action, not a restatement of the goal.
2. Keep each step imperative, checkable, and small enough to verify.
3. Mark the steps that stay on the user's side, such as logins or ethics decisions.
4. Name the handoff point to a neighboring skill when the work crosses a lifecycle boundary.

## Output

Return:

- The primary deliverable.
- The audit trail that lets a second reader check the work.
- Open items the skill could not resolve.

## Verification

- Write each line as a check that can fail, with a measurable condition.
- Separate what was verified in the current session from what the user must confirm.

## Safety

State what the skill must never do, what it must never store, and where professional judgment, ethics review, or legal advice replaces automation.

## Example prompt

"One realistic user request, in the language a user would actually type."

Expected smoke output:

- The shape of a correct first response, in two or three bullets.
